# Governance

This project begins in an explicit founder-led research phase.

That is not a flaw to hide. It is a condition to make visible.

## Current Governance Mode

Current mode: founder-led, public-reason-giving, contribution-seeking.

The founder may make final decisions about:

- repository direction,
- accepted contributions,
- document changes,
- issue priority,
- project framing,
- and whether a contribution becomes part of the public record.

The legitimacy requirement is that these powers become increasingly named, constrained, contestable, and eventually transferable where appropriate.

## Why Not Pretend To Be Decentralized

Premature decentralization can create confusion and hidden capture.

Early projects often need taste, coherence, and fast synthesis. Pretending that governance is already decentralized can be less honest than saying clearly:

> This project starts founder-led. The research question is how founder-led beginnings can move toward legitimate shared governance without theater.

## Founder Constraints To Explore

Candidate constraints:

- maintain a public decision log for major governance choices,
- distinguish editorial decisions from governance decisions,
- explain rejected substantive contributions,
- preserve accepted contribution records,
- publish phase-transition criteria,
- invite structured critique of founder powers,
- and define which powers can be transferred first.

## Phase Model

### Phase 0: Founding Research

Purpose:

Define claims, contribution standards, legitimacy schema, and first tests.

Governance:

Founder-led.

Exit condition:

The repo has a coherent claim ledger, contribution workflow, value rubric, threat model, and first contribution log.

### Phase 1: Public Contribution Pilot

Purpose:

Run the first real contribution workflow with external contributors.

Governance:

Founder-led with public review notes, appeal path, and monthly synthesis.

Exit condition:

At least several non-founder contributions are accepted, contested, or rejected with visible rationale.

### Phase 2: Review Pluralization

Purpose:

Introduce non-founder reviewers for specific contribution classes.

Governance:

Founder retains final authority but named reviewers participate in validation.

Exit condition:

Reviewer judgments are reliable enough to compare, contest, and audit.

### Phase 3: Limited Governance Transfer

Purpose:

Transfer specific powers, not vague "control."

Possible transferred powers:

- rubric revision,
- contribution taxonomy changes,
- review cadence updates,
- issue prioritization,
- appeal review.

Exit condition:

Transferred powers improve legitimacy without increasing capture risk.

### Phase 4: Institutional Generalization

Purpose:

Apply the pattern to other repos or public-goods contexts.

Governance:

To be determined by prior phases.

Exit condition:

Transfer conditions are documented and tested.

## Governance Research Questions

1. Which powers must remain centralized at the beginning?
2. Which powers can be transferred earliest?
3. What milestones are hard enough to prevent theater but reachable enough to create progress?
4. How can contributors contest governance without turning every decision into politics?
5. How can exit preserve contribution records and future claims?
