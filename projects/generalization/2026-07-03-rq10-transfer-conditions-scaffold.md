# RQ10 Transfer Conditions Scaffold

Status: non-adopted research scaffold.

This scaffold turns RQ10 into reviewable questions. It does not claim that this
repo has generalized, does not choose a second pilot, does not adopt governance
rules, does not change claim status, and does not create reward, rights,
sanction, disclosure, or public-posture commitments.

## Source Question

RQ10 asks what must be true before the pattern can apply beyond one repo.

The live claim boundary is C8 in `CLAIMS.md`: repo-scale success may generalize
to broader collaborative institutions, but this is a horizon claim whose proof
burden is transfer conditions and future comparative pilots.

## Transfer Object

The possible transfer object is not a full institution or a governance package.
At this stage, the transferable object is only a coupled design pattern:

```text
contribution evidence
  -> contestable validation
  -> visible contribution record
  -> value/rubric discussion
  -> rights/reward/governance questions held behind explicit gates
  -> capture monitoring as value accumulates
```

If that coupled pattern cannot survive outside the source repo without private
founder context, the generalization claim should stay horizon-only.

## Minimum Transfer Conditions

| condition | question | evidence needed before movement |
|---|---|---|
| Source-repo coherence | Has the first repo run contribution intake, review, log, contestability, and synthesis as one inspectable workflow? | Public contribution traces, review notes, log entries, and explicit unresolved residues. |
| Private-context reduction | Can a newcomer understand why a contribution was accepted, revised, or rejected without private founder explanation? | Rationale quality checks and at least one outsider-readable decision trace. |
| Context separation | Which parts are generic protocol shape, and which are local taste, domain knowledge, or founder judgment? | Component inventory with "portable", "adaptable", and "local-only" labels. |
| Legitimacy feedback | Do participants have visible paths for critique, correction, and exit? | Feedback channels, contested-decision traces, and unresolved-objection handling. |
| Capture pressure | Does added value make gaming, cartel behavior, or governance capture more attractive? | Attack-profitability review before any reward, rights, or governance transfer. |
| Governance load | Does the target setting need more process than the source repo can justify? | Cost/benefit notes and stop criteria for process drag or governance theater. |
| Reward delay | Can the pattern transfer without promising rewards before validation and capture controls are mature? | Explicit no-reward or reward-deferred baseline for the target context. |
| Public consequence boundary | Would transfer create legal, financial, reputational, or external-publishing consequences? | Joe/governance review before any external action or public adoption claim. |

## Candidate Context Types

These are context types for future comparison, not selected pilots.

| context type | why it matters | early transfer risk | first question to ask |
|---|---|---|---|
| Adjacent research repo | Closest match to the source-repo workflow. | Imports local process as ceremony. | Which contribution records are actually hard to judge today? |
| Open-source maintainer project | Has real maintenance, issue, and review pressure. | Treats contribution volume as value. | Can the log preserve non-code maintenance and review work? |
| Public-goods funding process | Directly tests value allocation and legitimacy claims. | Reward logic dominates before validation is ready. | Can review stay contestable without becoming a payout machine? |
| Local civic or cooperative project | Tests whether non-repo contribution can still be made inspectable. | Evidence and authority may become informal or personal. | What counts as inspectable contribution evidence outside GitHub? |

## Failure Signals

- The first repo still depends on founder intuition for ordinary contribution
  decisions.
- A target context requires reward logic, rights policy, sanctions, disclosure
  rules, or governance powers that this repo has not yet justified.
- Transfer would make public-posture claims stronger than the evidence.
- The target context cannot preserve contributor voice, exit, contestability, or
  visible rationale.
- Attack profitability increases because value accumulates faster than
  validation and capture monitoring.
- The governance package becomes workflow sediment rather than legitimacy
  improvement.

## Relationship To Existing Portability Work

`projects/governance-package-portability/` is a concrete case study about
whether governance machinery can be inventoried, adapted, and tested.

This RQ10 scaffold is broader and should not conclude that the governance
package is portable. It can use that project later as evidence if the inventory,
origin study, port review, and kill criteria become strong enough.

## Review Questions

Before RQ10 moves beyond a scaffold, a reviewer should be able to answer:

1. What exactly is being transferred?
2. Which source-repo traces show the pattern working?
3. Which pieces are local-only?
4. What would count as a failed transfer?
5. Which target contexts are excluded until reward, rights, governance, and
   attack-modeling gates clear?
6. What public language keeps C8 horizon-only until comparative evidence exists?

## Stop Conditions

Stop and require Joe/governance review before:

- naming or committing to a pilot target;
- claiming this project has generalized;
- changing C8 status;
- adopting transfer criteria as policy;
- importing governance-package machinery into another repo;
- making reward, rights, sanction, disclosure, governance, or public-posture
  commitments;
- publishing, submitting, deploying, or externally announcing any transfer.
