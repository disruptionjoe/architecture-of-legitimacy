# RQ10 First-Pilot Component Inventory

Status: draft component-inventory scaffold, not adopted transfer policy, C8
movement, portability claim, public posture, target selection, governance,
rights, reward, AI policy, or contribution-record policy.

This inventory classifies pieces of the synthetic first-pilot stack so later
review can ask what might transfer, what would need adaptation, what is local
only, and what is still unknown. It does not claim that any component is
portable in practice, choose a target context, export governance-package
machinery, launch a pilot, invite contributors, or create reward, rights,
governance, sanction, disclosure, portability, public-posture, or
external-consequence commitments.

## Purpose

The RQ10 first-pilot transfer screen names a safe next local move:

```text
classify one first-pilot component inventory as portable candidate,
adaptable, local-only, or unknown
```

This file performs that inventory for the current synthetic first-pilot stack.
The useful output is not a transfer plan. The useful output is a clearer stop
map:

```text
which parts of the first-pilot stack can be discussed as possible pattern
components, and which parts must stay tied to this repo, this evidence base,
or later governed review
```

## Classification Labels

| label | meaning in this inventory | forbidden inference |
|---|---|---|
| `portable_candidate` | The component may be discussable as a general pattern after source-repo traces and target evidence mature. | It is not proven portable, ready to export, or adopted outside this repo. |
| `adaptable` | The component may have a reusable shape, but a target context would need renamed fields, evidence rules, or governance review. | The current repo wording should not be copied directly into another context. |
| `local_only` | The component depends on this repo's research agenda, current scaffolds, founder context, or governance boundaries. | It should not be represented as transferable infrastructure. |
| `unknown` | The component cannot be classified until live source traces, target evidence, or second-ring decisions exist. | Unknown does not mean blocked forever, safe, or portable. |

These labels are review-prep labels only. They are not contribution states,
review states, transfer states, or public workflow labels.

## Minimum Evidence Before Reclassification

A later reviewer should not upgrade any `adaptable` or `unknown` component
toward a stronger transfer claim until the packet can show:

1. at least one source-repo live trace for accepted, revised, rejected, and
   contested contribution paths;
2. a target-specific evidence packet that names public, private, and unsafe
   evidence;
3. a local-only component inventory for the target context;
4. reward and rights deferral language that the target can actually preserve;
5. a capture-pressure read specific to the target's value, authority, and
   visibility;
6. an AI-assistance boundary suitable for the target's tools and records;
7. a public-language boundary that keeps C8 horizon-only;
8. and an abort criterion that would reject transfer before launch.

Until those exist, the safest classification for the whole stack remains:

```text
useful source-repo review preparation, not transfer evidence
```

## Target-Context Non-Selection

This inventory intentionally does not choose a target context. It also does not
rank context types. The classifications above are only about the current
source-repo stack.

If a later governed review considers a target, it should create a new
target-specific packet before using any label from this file. That packet would
need to rename the component, identify target evidence, list target-specific
protected surfaces, and state why the component should be abandoned if the
target cannot preserve the boundary.

## Safe Local Consequence

This inventory allows one narrow conclusion:

```text
The first-pilot stack now has a visible component map for later RQ10 review,
but the map classifies transfer risk rather than proving transfer readiness.
```

That distinction makes it harder for a coherent synthetic packet to turn into
portability language by implication.

## Review Questions

Before any later RQ10 packet treats this inventory as useful evidence, a
reviewer should answer:

1. Which components have live source-repo traces rather than synthetic traces?
2. Which labels would change if the target has different evidence norms,
   contributor expectations, or public consequences?
3. Which component carries the largest overclaim risk if copied as-is?
4. Which component depends most on founder judgment or local research taste?
5. Which reward, rights, capture, governance, or AI gate would fail first in a
   target context?
6. Which public words are still forbidden because C8 remains horizon-only?
7. What would force the repo to abandon transfer even after local pilot success?
