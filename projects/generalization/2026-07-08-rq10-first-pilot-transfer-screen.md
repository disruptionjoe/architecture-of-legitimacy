# RQ10 First-Pilot Transfer Screen

Status: draft review-prep scaffold, not adopted transfer policy, C8 movement,
public posture, pilot selection, governance, rights, reward, AI, or
contribution-record policy.

This screen applies the existing RQ10 transfer scaffolds to the synthetic RQ5
first-pilot packet. It does not claim that the repo has generalized, choose a
second repo or target context, export governance-package machinery, change
transfer language, change C8 status, launch a pilot, invite contributors, or
create reward, rights, governance, sanction, disclosure, portability,
public-posture, or external-consequence commitments.

## Purpose

The first-pilot stack now has enough local review-prep structure to make a
transfer hazard visible:

```text
If one synthetic first-pilot packet looks coherent, what must prevent that
coherence from being described as portability, transfer readiness, or proof
that the workflow generalizes?
```

The answer should be conservative. A coherent first-pilot packet can improve
source-repo review preparation. It cannot, by itself, make C8 stronger or
select a target context.

## Target-Context Non-Selection

This screen intentionally does not choose a target. It names why each context
type remains blocked before selection.

| context type | why it remains blocked now | evidence needed before reopening |
|---|---|---|
| Adjacent research repo | Source-repo live traces are missing, and local research standards may not be portable. | Accepted, revised, rejected, and contested traces plus a local-only component inventory. |
| Open-source maintainer project | Volume, maintainer authority, and non-code review pressure could make rationale ceremonial. | A bounded trace comparison with reviewer-load and authority limits. |
| Public-goods funding process | Money converts weak contribution review into a high-value attack surface. | Target-specific RQ9 reward readiness and RQ8 capture pressure before any discussion. |
| Local civic or cooperative project | Civic or cooperative use can create public power, representation, or reputational effects. | Joe/governed review plus privacy-safe evidence and external-consequence review. |
| Governance-package portability case | Portability inventory is not proof that this research pattern generalized. | Origin inventory, adaptation boundary, kill criteria, and process-drag evidence. |
| Regulated or confidential setting | Visible rationale can conflict with protected evidence. | Safe substitute-evidence design before a target is even considered. |
| High-volume contribution environment | The workflow could add ceremony faster than legitimacy. | Cost, sampling, escalation, and failure-rate evidence. |

## Safe Local Consequence

This screen allows one narrow conclusion:

```text
The first-pilot stack can now show where transfer overclaim would happen.
It still cannot supply transfer evidence.
```

That distinction is useful because it lets later review inspect transfer risk
without making the transfer move.

## Review Questions

Before any later RQ10 discussion treats first-pilot work as relevant evidence,
a reviewer should be able to answer:

1. Which source-repo traces are real rather than synthetic?
2. Which packet fields are genuinely portable, and which are local-only?
3. What target-context evidence would be public, private, or unsafe to expose?
4. Which second-ring gate would fail first in the target context?
5. What public-language boundary keeps C8 horizon-only?
6. What evidence would force the repo to reject transfer even after a successful
   local first pilot?
