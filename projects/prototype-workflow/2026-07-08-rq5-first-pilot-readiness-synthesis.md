# RQ5 First-Pilot Readiness Synthesis

Status: draft review-prep synthesis, not adopted workflow, review, governance,
rights, reward, AI, transfer, public-posture, claim-status, test-status, or
contribution-record policy.

This synthesis consolidates the separate first-pilot screens now attached to
the synthetic RQ5 packet. It does not launch a pilot, select a real target,
choose a live record surface, invite public contributions, open an issue or PR,
edit templates or standards, change review policy, assign reviewer authority,
create a live contribution-log entry, mark tests passed, move claims, create
rights or rewards, change governance, change AI policy, make transfer claims,
change public posture, or decide any real contribution.

## Purpose

The repo now has many local screens for the same synthetic first-pilot packet:
eligibility, class/evidence, value, rationale, legitimacy failures,
log-readiness, governance, rights, capture, reward, transfer, and AI support.

The remaining RQ5 question is not whether the first pilot can run. It cannot run
from these artifacts alone. The useful question is narrower:

```text
What does the current packet stack make ready for governed review, and what
must still be blocked before live use?
```

This file is a compact ledger for that boundary.

## What Is Ready

The packet stack is ready for a governed review-prep conversation about:

1. whether a bounded source-backed research/review contribution remains the
   safest first-pilot shape;
2. what the live packet would have to preserve before review;
3. which missing source, target, mechanism, and limitation fields block review;
4. which live choices require Joe or governed authorization;
5. how a later packet should preserve boundaries across eligibility, class,
   value, rationale, loss notes, log eligibility, contestability, governance,
   rights, capture, reward, AI, and transfer.

This is readiness for review of the packet shape, not readiness to run a pilot.

## What Remains Blocked

These actions remain blocked from unattended Progress and require Joe or
governed review:

- selecting a real first-pilot target or contributor;
- choosing an issue, PR, project artifact, or contribution-log surface for live
  use;
- inviting public contributions;
- editing issue templates, contribution templates, contribution standards, or
  the contribution-log schema;
- assigning reviewer authority or adopting review cadence, appeal, dispute, or
  response-time policy;
- creating a live `CONTRIB-*` entry;
- changing claim status, test status, public posture, governance, legitimacy
  conditions, rights, rewards, AI policy, or transfer posture;
- treating synthetic screens as evidence about real contributor behavior;
- or taking any non-GitHub external action.

## Minimum Governed Review Packet

Before a later live pilot can be considered, a governed review packet should at
least answer:

1. Which public target is under review?
2. What is the one contribution unit?
3. Which stable sources support the submitted facts?
4. Which claims are source-backed, and which are inference?
5. Which primary class and secondary effects apply, and why?
6. What evidence minimum is still missing?
7. What value dimensions can be discussed without scores or reward meaning?
8. What decisive rationale and protected boundary would be visible?
9. What loss notes or useful residue survive without promise language?
10. Why is the packet log-eligible or not log-eligible after review?
11. What exact point is contestable without adopting appeal rights?
12. Which second-ring stop fires first: governance, rights, capture, reward,
    AI, transfer, public posture, or external consequence?

If those questions cannot be answered from public, reviewable records, the next
state remains `source_packet_needed`, not live pilot readiness.
