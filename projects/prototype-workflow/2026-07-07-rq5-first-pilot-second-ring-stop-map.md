# RQ5 First-Pilot Second-Ring Stop Map

Status: draft review-prep scaffold, not adopted workflow, governance, rights,
reward, AI, transfer, public-posture, claim-status, or contribution-record
policy.

This stop map connects the first-pilot review packet checklist to the
second-ring dependency map. It names where a later human or governed first-pilot
review must pause, narrow, or route for explicit decision before unresolved
second-ring questions become real institutional commitments.

It does not invite a public pilot, choose a pilot target, edit issue templates,
change contribution standards, create a live contribution-log entry, adopt
review policy, assign reviewer authority, mark tests passed, move claim status,
change governance, create participant rights, change reward or retained-value
meaning, change AI policy, make transfer claims, change public posture, or
decide any real contribution.

## Purpose

The first-ring scaffolds now make a later first pilot more inspectable: a packet
can preserve contribution unit, evidence, review rationale, useful residue, log
eligibility, and contestability without relying entirely on private founder
context.

That is still not enough to run or advertise a real pilot. A first pilot can
quietly cross second-ring boundaries if a review packet starts to imply rights,
reward, governance authority, capture response, AI authority, or transfer
claims. This file is the pause layer between "the packet can be described" and
"the packet may be used live."

The review question is:

```text
Which unresolved second-ring dependency would turn a narrow first-pilot review
into policy, rights, reward, governance, AI authority, transfer, or public
posture if the packet proceeded without an explicit stop?
```

## How To Use This Map

Use this map after a later first-pilot packet has passed the first-ring
continuity checks. It asks whether the packet can remain a narrow contribution
review, or whether it has touched a second-ring dependency that requires a stop.

The safe output of this map is one of three dispositions:

| disposition | meaning | allowed next move |
|---|---|---|
| Continue narrowly | The packet can stay inside a low-governance review lane. | Record the boundary and continue review-prep without policy adoption. |
| Narrow before review | The packet contains useful material but one consequence should be removed from scope. | Rewrite the packet boundary or route the risky part to a later scaffold. |
| Stop for Joe/governed review | The next step would create or imply protected institutional meaning. | Do not proceed live; make the decision explicit before any pilot action. |

These dispositions are review-prep labels, not active workflow states.

## Second-Ring Stop Map

| dependency | first-pilot trigger | stop or narrow if | safe review-prep move | explicit review point |
|---|---|---|---|---|
| RQ6 governance transition | A review packet needs founder exception, reviewer assignment, emergency handling, rollback, or rule interpretation to decide the contribution. | The packet would define who has authority, which power is transferred, when an exception expires, or how rollback works. | Name the unresolved authority dependency and keep the contribution outcome provisional or synthetic. | Founder powers, emergency rules, rollback rules, reviewer authority, and phase movement. |
| RQ7 voice, exit, and retained value | The packet contains identity change, contributor departure, adverse record, useful residue, or contestability pressure. | Loss notes, retained markers, correction paths, or public records would imply rights, erasure, appeal, privacy, retained value, or governance standing. | Preserve the narrow rationale and useful residue without promise language. | Participant rights, exit rules, identity policy, appeal/dispute paths, and retained-value meaning. |
| RQ8 attack and capture modeling | The packet raises conflict, sponsor pressure, volume gaming, rubric capture, synthetic consensus, or prior-art shallowness. | The response would reject, sanction, limit, disclose, identify, or publicly label a real contributor; or it would change eligibility, review, governance, reward, or public posture. | Add an observability note: attack pattern, variable under pressure, local repair, and escalation boundary. | Disclosure rules, sanctions, contributor limits, capture-response policy, and public posture. |
| C7/T8 AI authority boundary | The packet uses AI assistance for drafting, synthesis, scoring, source review, or rationale preparation. | AI output becomes the final review reason, hidden evidence, synthetic consensus, score substitute, novelty judgment, or policy basis. | Require human-owned rationale, source trail, and visible AI role boundaries in the packet. | AI-use policy, reviewer authority, evidence policy, and automation limits. |
| RQ9 reward logic | The packet mentions score, value, retained contribution, future recognition, payout analogy, token-like language, or governance weight. | Scores become balances, loss notes become payout debt, retained markers imply rights, or acceptance implies reward readiness. | State the no-reward baseline and separate record, recognition, reward, and future-review markers. | Reward mechanisms, score conversion, payout formulas, retained-value rights, and governance weight. |
| RQ10 generalization | The packet is framed as proof that the workflow works beyond this repo or can transfer to another context. | The review would choose a second repo, make a public transfer claim, export process language, or create external commitments before local traces are mature. | Keep the finding local and horizon-only; record what would need target-context evidence later. | Transfer pilots, public generalization language, portability commitments, and external consequences. |

## Cross-Cutting Stop Conditions

A later first-pilot review should stop before live use if any of these become
necessary to interpret the packet:

- public invitation language;
- a selected live pilot target;
- issue-template, contribution-template, or contribution-standard edits;
- live `CONTRIB-*` entry before review is complete;
- review cadence, appeal, dispute, reviewer-authority, or response-time policy;
- claim-status movement or test pass/fail marking;
- reward, retained-value, legal, financial, rights, ownership, employment, or
  governance commitments;
- sanctions, contributor limits, disclosure requirements, identity rules, or
  capture-response policy;
- AI as final authority, hidden evidence, or score substitute;
- transfer claims, candidate second repo selection, or public proof language;
- or any non-GitHub external consequence.

## Minimum Stop-Screen Fields

If a later first-pilot packet invokes this map, the packet should record these
fields before a human or governed review decides whether to proceed:

| field | question |
|---|---|
| Dependency invoked | Which second-ring question did the packet touch? |
| Live consequence risk | What institutional meaning could be created by proceeding? |
| Safe narrow interpretation | What can still be reviewed without crossing the boundary? |
| Blocked next action | Which concrete action must not happen yet? |
| Review owner | Does this require Joe, repo governance, or a later scaffold? |
| Non-conclusion | Which claim, test, policy, right, reward, governance, AI, transfer, or public-posture move was not made? |

These fields should not be promoted into a required contribution template
without a separate governed decision.

## Example Review-Prep Read

A bounded source-backed research note could still be a good first-pilot shape.
This map does not change that. It says the note should stop or narrow if its
review requires any of these interpretations:

- accepting the note would change claim status or public novelty language;
- rejecting the note would create sanctions, eligibility rules, or identity
  conclusions;
- preserving useful residue would imply future payout, retained rights, or
  governance standing;
- AI-generated synthesis would become the decisive review reason;
- sponsor or conflict context would require a new disclosure rule;
- or the packet would be described as proof that the workflow can transfer
  beyond this repo.

The same note can remain reviewable if the packet keeps those effects out of
scope and records them as explicit non-conclusions.

## Relationship To The First-Pilot Checklist

The first-pilot review packet checklist answers:

```text
What fields must a later packet preserve?
```

This stop map answers:

```text
When does preserving those fields become unsafe because a second-ring dependency
would be decided by implication?
```

The stop map therefore sits after packet completeness and before live pilot
use. It is not a replacement for the checklist and does not create a live pilot
process.
