## Working Distinctions

These distinctions are review scaffolding, not adopted policy.

| term | working meaning for later T9 review | common false equivalence |
|---|---|---|
| Retained record | A durable trace that work happened, who or what it was attributed to, what process touched it, and what state it reached. | "The record exists, therefore future value was preserved." |
| Retained recognition | Reputation, authorship credit, profile history, edit attribution, reviewer service credit, or other status attached to the durable record. | "Recognition is the same as a reward or right." |
| Retained reward | A concrete allocation such as money, tokens, funding, publication benefit, or other material outcome granted because the work counted. | "Reward means the system preserved all legitimacy-relevant value." |
| Retained-value marker | A non-promissory or promissory marker that explicitly preserves possible future relevance, future review eligibility, or a deferred claim. | "Any durable trace is already a retained-value marker." |

The retained-record lane should not ask only whether a neighboring system keeps
history. It should ask what the history does later: nothing, recognition,
allocation, eligibility, control, review priority, dispute evidence, or a
stronger claim.

## Local Comparison Pressure

The prepared packets suggest that "retained value" is too broad for a later
collision review unless it is decomposed.

1. Funding systems can retain reward without preserving contribution-level
   adverse-decision trace.
2. Knowledge-validation systems can retain detailed record and recognition
   without granting property-like rights, monetary reward, or explicit future
   eligibility.
3. Accepted-path records are much easier to find than losing, rejected,
   filtered, reverted, or under-valued paths.
4. A profile, publication, or edit history may preserve reputation while still
   failing to preserve the specific local burden: visible relation between
   contribution evidence, legitimacy rationale, loss notes, and any future
   value marker.

For later T9 review, a neighboring system should not be treated as satisfying
the retained-value burden merely because it has any one of:

- public history;
- attribution;
- reputation;
- reward allocation;
- project or author profile;
- public results page;
- publication record;
- edit history;
- appeal or dispute path.

The review should name which one is present and which one is absent.

## Possible Review Questions

These are candidate questions for a later verdict-bearing T9 packet, not
decisions.

1. Does the system preserve the adverse path, or only the accepted/funded path?
2. Is the retained trace attached to a contribution, a contributor, a project,
   a manuscript, a page, a round, or an institution?
3. Does the trace create no later effect, reputation, funding, governance
   standing, review priority, or legal/financial entitlement?
4. Can a future reviewer reconstruct why value was lost, narrowed, rejected,
   revised, reverted, filtered, or under-rewarded?
5. Does the system distinguish retained record from retained reward in its own
   process language?
6. If recognition later affects allocation or governance, is that effect
   explicit, contestable, and bounded?

## Non-Promissory Local Language Test

The RQ7 boundary map offers a useful later language test:

```text
This record may matter for later review, but it creates no payout, ownership,
legal claim, governance weight, guaranteed review priority, or reward promise.
```

That sentence is not adopted policy. It is a stress test. If a local retained
value marker cannot be described with that kind of negative boundary, it is
probably no longer merely a retained-record clarification; it has crossed into
rights, reward, governance, or legal/financial policy and needs review.

## Implication For T9 Lane Selection

The retained-record lane is now narrower:

```text
Compare how neighboring systems preserve accepted and adverse contribution
paths, and whether the preserved record later does any work beyond recognition.
```

This makes the next T9 step smaller than a full collision verdict. A later
review could choose one neighboring system and one path type:

- Optimism: under-rewarded or non-selected application path.
- Gitcoin: rejected, low-funded, or sybil-flagged application path.
- Wikipedia: reverted edit or disputed contribution path.
- Academic peer review: desk-rejected, rejected-after-review, or revised
  manuscript path.

The target evidence would be not "does the system have records?" but:

- what record survives;
- who can inspect it;
- what rationale is visible;
- what value or recognition remains;
- what future use is allowed;
- and what is explicitly not promised.
