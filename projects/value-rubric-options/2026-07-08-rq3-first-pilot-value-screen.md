# RQ3 First-Pilot Value Screen

Status: draft review-prep scaffold, not adopted rubric policy, score policy,
review policy, reward policy, governance policy, claim-status movement,
test-status movement, public-posture change, or contribution-record policy.

This screen applies the RQ3/T2 value-rubric scaffolds to the synthetic RQ5
first-pilot packet. It does not score a real contribution, rank contributors,
adopt a rubric option, change score meanings, assign reviewer authority,
convert scores to reward, change `CONTRIBUTION-STANDARDS.md`, mark T2 or RQ5
passed, launch a pilot, create a live record, or decide any real contribution.

## Purpose

The RQ5 first-pilot packet makes a bounded source-backed research/review
contribution shape inspectable. The RQ3/T2 scaffolds explain how value scoring
can fail when reviewability, acceptance, reward, and governance are collapsed.

This file asks a narrower question:

```text
What value information can a first-pilot packet preserve before any score,
reward, ranking, or policy meaning is allowed?
```

The useful result is a non-scoring screen that helps a later reviewer separate
evidence gaps, dimension disagreement, governance stops, and reward leakage.

## Boundary Under Review

The current provisional rubric names seven dimensions:

- `clarity`
- `rigor`
- `relevance`
- `novelty`
- `legitimacy`
- `capture resistance`
- `coordination value`

The synthetic first-pilot packet is not scoreable because it has no real
contributor, no selected live target, no source set, no mechanism comparison,
and no real review decision. That is a feature of this screen, not a defect.

The first-pilot value question is not "what score should this get?" The safer
question is:

```text
Which value dimensions would need visible evidence before a later live packet
could be discussed without turning scores into authority or reward?
```

## First-Pilot Value Screen

| screen field | current synthetic answer | value risk if skipped | safe local output |
|---|---|---|---|
| Reviewability before scoring | The packet shape is coherent, but target, sources, and mechanism comparison are missing. | Reviewers fill missing evidence with private context or taste. | Record `not_scoreable_yet` as a prep note, not a review state. |
| Primary class | Likely `research`, with `review` as a secondary effect if the later note critiques an existing scaffold. | Class shifts silently to make scoring easier. | Preserve the class/evidence question before any value discussion. |
| Evidence minimum | Stable references, relevance target, source/inference split, mechanism comparison, and limitation are required later. | Link lists or confident summaries get relevance and rigor credit without reviewable support. | Name missing evidence fields explicitly. |
| Dimension prompts | The seven dimensions can frame questions, but they should not produce a score vector here. | Numeric scores imply precision and adoption readiness. | Use dimension-specific prompts only. |
| Disagreement type | Current likely type is `evidence gap`; later packets may expose `dimension boundary`, `governance stop`, or `reward leakage`. | Reviewer disagreement becomes arbitrary score debate. | Classify the disagreement before arguing about totals. |
| Review-state separation | `needs_revision` in the synthetic packet is a review-prep result, not a live review state. | A draft prep outcome becomes real contributor feedback. | Keep state labels visibly synthetic until live review is authorized. |
| Reward and governance stop | No score can imply payout, retained value, governance weight, authority, or future claim. | Value language becomes entitlement or power language. | Add a boundary note beside any value rationale. |

## Dimension Prompts For The First Pilot

These prompts are allowed as review-prep questions. They are not scores.

| dimension | first-pilot prompt | current synthetic signal | false positive to avoid | protected boundary |
|---|---|---|---|---|
| `clarity` | What target, evidence, and review object become easier to inspect? | The packet shape is clearer than an open-ended pilot invitation. | Treating polished prose as value without a selected target. | Do not accept or log a contribution for clarity alone. |
| `rigor` | Which source, assumption, limitation, or inference boundary is visible? | The required fields are named, but no source evidence is supplied. | Treating the field list as if it were evidence. | Do not score rigor until sources and limits exist. |
| `relevance` | Which live claim, test, research question, or scaffold would the packet affect? | RQ5/T3/T10/T11 are likely, with possible T9 adjacency. | Allowing broad topical fit to substitute for a target. | Do not choose the live pilot target here. |
| `novelty` | What distinction or mechanism would the repo miss without the contribution? | Unknown until a real neighboring system or mechanism is selected. | Rewarding unusual framing before evidence exists. | Do not make a novelty or prior-art verdict. |
| `legitimacy` | Can a later reader reconstruct, question, or contest the value rationale? | The packet preserves rationale, loss-note, and contestability fields. | Treating fairness language as legitimacy without a visible record. | Do not create appeal rights, dispute process, or reviewer authority. |
| `capture resistance` | What gaming, conflict, AI-volume, or authority path becomes more visible? | The packet can require source/inference separation and boundary notes. | Scoring generic safety concern as mitigation. | Do not adopt disclosure, sanction, contributor-limit, or monitoring policy. |
| `coordination value` | What future review action becomes easier and less founder-dependent? | A later reviewer can see which fields must be filled before scoring. | Treating additional paperwork as coordination value. | Do not turn this screen into a mandatory template. |

## Disagreement Handling

A first-pilot value note should preserve the type of disagreement before any
score vector is considered.

| disagreement type | first-pilot example | useful if | stop boundary |
|---|---|---|---|
| Evidence gap | Reviewers cannot discuss `rigor`, `relevance`, or `novelty` because the source set is missing. | The missing source, target, mechanism, or limitation can be named. | Do not infer value from founder context. |
| Dimension boundary | The same observation could count as `legitimacy`, `capture resistance`, or `coordination value`. | The record names which dimensions compete and why. | Do not double-count one value signal as multiple scores. |
| Class sensitivity | A source-backed note may need different prompts than maintenance, implementation, or protocol design. | The class-specific evidence minimum becomes clearer. | Do not rewrite contribution classes or standards here. |
| Governance stop | A useful protocol-design observation would imply authority, rights, reward, or phase movement if adopted. | The useful content is preserved while the authority move is deferred. | Do not use value language to adopt governance. |
| Reward leakage | A reviewer starts treating high value as payout, retained value, ownership, or future claim. | The score meaning is restated as discussion-only. | Do not create reward, retained-value, legal, financial, or governance meaning. |
| Taste conflict | Reviewers disagree without naming evidence, dimension, class, or boundary. | The note asks what evidence would make the reason inspectable. | Do not launder preference as rubric judgment. |

## Non-Scoring Review Note Shape

A later live packet could include a value note shaped like this before any
score vector is allowed:

```yaml
value_reviewability: not_scoreable_yet | score_discussion_ready
primary_class_under_review: research
target_surface: <claim, test, research question, or scaffold>
evidence_minimum_status:
  source_identity: missing | present
  stable_references: missing | present
  mechanism_comparison: missing | present
  source_inference_split: missing | present
  limitation_statement: missing | present
dimension_prompts_discussed:
  - clarity
  - rigor
  - relevance
primary_disagreement_type: evidence_gap
value_preserved_without_adoption: <one sentence>
protected_boundary: >
  This note does not score, rank, reward, accept, log, govern, change claims,
  change tests, assign authority, or create contribution-record meaning.
```

This shape is a prompt for later review design. It is not a live template,
active issue form, contribution-log schema, review policy, or rubric policy.

## What This Prepares

This screen prepares a later human/governed first-pilot discussion by making
these distinctions explicit:

1. reviewability before value scoring;
2. dimension prompt before numeric score;
3. disagreement type before score totals;
4. value preservation before acceptance;
5. useful residue before reward or retained-value meaning;
6. governance stop before authority movement;
7. synthetic prep state before real contributor feedback.
