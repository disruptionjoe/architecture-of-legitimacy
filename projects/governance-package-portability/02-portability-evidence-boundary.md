# Governance Package Portability Evidence Boundary

Status: evidence boundary; no portability verdict assigned.

Purpose: define what evidence a later governance-package portability study must
preserve before treating any institutional component as portable, adaptable, or
unsafe to port.

repository, does not evaluate another repository, does not assign a portability
class to any component, does not change transfer policy, does not change public
posture, and does not create rights, reward, governance, contributor, or

## Why This Boundary Is Needed

The existing inventory names plausible governance-package components: steward
roles, run records, claim ledgers, kill criteria, promotion gates, memory,
cadence, personas, absorber tests, and governance-change ledgers.

Those labels are not evidence. A later portability study must show that a
component solved a real institutional problem in its origin setting, survived
or changed under transfer pressure, and fits the brownfield target without
duplicating or weakening native governance.

The risk is not only copying too much process. It is creating false legitimacy:
the appearance that a repository has stronger review, claim discipline,
contestability, or portability just because it has inherited governance
surfaces.

## Evidence Objects

Later study should separate four evidence objects before making any component
judgment.

| evidence object | review question | failure if missing |
|---|---|---|
| Origin function evidence | What did the component do in actual work that would not have happened as well without it? | Component presence is mistaken for component value. |
| Transfer behavior evidence | What changed, broke, stayed useful, or became stale when the component moved outside its origin setting? | Copying is mistaken for portability. |
| Brownfield conflict evidence | Which native target-repo surfaces already solve the same problem or would be displaced? | Porting creates duplicate authority or process drag. |
| Legitimacy effect evidence | Did the component improve visibility, contestability, bounded authority, non-capture, claim discipline, or evidence quality? | The component becomes ceremony without legitimacy gain. |

## Source Versus Inference Boundary

Each later evidence packet should mark rows this way:

| label | meaning | allowed use |
|---|---|---|
| `source_fact` | Directly visible in a cited repo artifact, run record, decision record, or public process source. | May support a narrow factual statement. |
| `observed_effect` | A change visible across work outputs, such as repeated use, avoided duplication, clarified blocker, or stopped overclaim. | May support cautious function evidence if linked to source facts. |
| `reviewer_inference` | The reviewer believes the component caused or helped an effect. | Must stay contestable and cannot become a portability verdict alone. |
| `unsupported_hypothesis` | A plausible but unproven portability claim. | May become a next research question, not a finding. |
| `stop_boundary` | A point where adoption, external consequence, policy, rights, rewards, or public posture would be implicated. | Requires Joe or repo-governed review before action. |

## Component Evidence Tests

Use these tests before assigning any tentative portability class.

| component class | minimum evidence before classification | stop boundary |
|---|---|---|
| Anchored steward | At least one origin or transfer run where steward rules changed selection, prevented drift, preserved authority, or avoided overclaim. | Do not create or replace a target steward role from this study. |
| Run records and closeout | Evidence that a later run actually used prior records to avoid duplication, resume work, or improve a next decision. | Do not impose new receipt fields on a target repo. |
| Claim ledger and status vocabulary | A visible claim movement, paused movement, or prevented overclaim tied to a status rule. | Do not change claim status or canon posture. |
| Kill criteria and path-kill records | A killed, paused, or resurrected path with explicit conditions and later use. | Do not mark any target path killed. |
| Promotion gates | A claim, workflow, role, or artifact delayed, narrowed, or rejected because a gate applied. | Do not promote or demote any target artifact. |
| Memory summary and append-only memory | Evidence that memory reduced repeated context loading or preserved a decision without becoming shadow canon. | Do not create memory surfaces in another repo. |
| Next-trigger plan and cadence | Evidence that cadence selected worthy work without pressure to invent work. | Do not schedule or recommend cadence changes. |
| Persona registry and adversarial review | A review where named perspectives found materially different failures and led to better synthesis. | Do not add persona machinery to a target repo. |
| Absorber discipline and same-neighbor-data tests | A comparison where the test narrowed overclaim or clarified non-absorption. | Do not make novelty or collision verdicts. |
| Governance-change ledger | A process change whose reason, authority, and later effect remained reviewable. | Do not recommend a governance-change ledger for another repo. |

## Brownfield Conflict Checks

Before any GU or other brownfield port plan, the study should ask:

| conflict surface | check |
|---|---|
| Existing canon/status model | Would a new claim ledger clarify status, or create competing truth surfaces? |
| Existing process records | Would new run records preserve missing accountability, or duplicate current progress history? |
| Existing review/persona machinery | Would personas add adversarial coverage, or rename review work already happening? |
| Existing specification protocol | Would imported promotion gates strengthen native admission rules, or weaken the target vocabulary? |
| Existing research cadence | Would scheduled progress produce better work, or create pressure to manufacture activity? |
| Existing memory/status files | Would a memory pack reduce context load, or become stale shadow authority? |

## Minimum Later Evidence Packet Shape

A later packet should use one row per component:

| field | required content |
|---|---|
| Component | Candidate governance-package component. |
| Institutional problem | Specific problem the component is supposed to solve. |
| Origin source facts | Cited origin artifacts and run evidence. |
| Origin observed effects | What actual work changed because the component existed. |
| Transfer source facts | Cited transfer artifacts and run evidence. |
| Transfer observed effects | What survived, adapted, failed, or became stale. |
| Brownfield conflict surface | Native target surfaces that overlap or conflict. |
| Legitimacy effect | Visibility, contestability, bounded authority, non-capture, claim discipline, or evidence-quality effect. |
| Reviewer inference | Cautious interpretation separated from facts. |
| Tentative class | Universal, research-program core, optional enhancement, target-specific, do-not-port, or unresolved. |
| Kill condition | Evidence that should defer, reject, or remove the component. |
| Stop boundary | Any adoption, public posture, rights, reward, governance, or cross-repo action boundary. |

## Candidate Next Review Shape

The safest next material study is an origin-function evidence packet for two or
three components only, not a whole-package verdict. Good first candidates are:

1. run records and closeout;
2. claim ledger and status vocabulary;
3. kill criteria and path-kill records.

These are high leverage because they can be tested against actual work records
without recommending a target repo port.
