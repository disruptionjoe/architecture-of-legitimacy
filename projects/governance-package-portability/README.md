# Governance Package Portability Study

Status: initial scoping

## Purpose

This project studies whether governance machinery that emerged inside
`temporal-issuance` can be understood as a reusable institutional technology
for AI-assisted research repositories.

The immediate question is not whether the package should be copied into
another repository. The immediate question is whether the governance package
itself can be inventoried, evaluated, adapted, and tested without creating
process drag or false legitimacy.

## Why This Belongs Here

`architecture-of-legitimacy` studies contribution allocation, validation,
review, contribution logs, governance transitions, capture resistance, and
protocol design. The governance-package portability question is a concrete
case study of those themes:

- contribution and claim records;
- review cadences and run logs;
- legitimacy of agent-produced research outputs;
- capture and overclaim resistance;
- governance transition from one research context to another;
- reusable institutional design for AI-native research.

## Working Object

The "governance package" is provisionally understood to include:

- anchored repo steward pattern;
- run protocols, run logs, run closeout, and run receipts;
- claim ledgers and status vocabularies;
- kill criteria, path-kill records, and resurrection triggers;
- memory summaries, append-only memory logs, and export queues;
- promotion gates for claims, workflows, and personas;
- persona registries and multi-persona review workflows;
- absorber discipline and same-neighbor-data tests;
- next-trigger plans, cadence rules, and learning-return schemas;
- governance-change ledgers and stewardship metrics.

This list is not yet a portability claim. It is the first inventory target.

## What Is Not Yet Known

- Which components are actually central in `temporal-issuance`.
- Which components transferred cleanly into `ai-native-epistemic-systems`.
- Which transferred components created overhead, stale surfaces, or governance
  theater.
- Whether `gu-formalization` needs any of this package, given that it already
  has process, canon, exploration, status-label, and specification surfaces.
- Whether a brownfield port should be a transplant, a thin adapter, or no port.

## Planned Outputs

This project will produce:

1. a governance package inventory;
2. a portability evidence-boundary map;
3. a Temporal Issuance origin study;
4. an AI-Native Epistemic Systems port review;
5. a first-pass portability model;

8. review questions and open questions for the portability thesis.

## Success And Failure

Success means producing a grounded model of what is reusable, what requires
domain adaptation, what should not be ported, and what evidence would show that
a port improved or harmed a research repo.

Failure means the study cannot distinguish useful governance machinery from
local workflow sediment, or it recommends copying process into GU without
evidence that the process solves a GU-specific problem.

The project should be willing to conclude that the package is only partially
portable, not portable, or useful only as a diagnostic lens.
