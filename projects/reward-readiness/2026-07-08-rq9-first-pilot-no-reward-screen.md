# RQ9 First-Pilot No-Reward Screen

Status: draft review-prep scaffold, not adopted reward logic, reward policy,
score conversion, retained-value right, governance weight, public posture,
claim-status, test-status, or contribution-record policy.

This screen applies the current RQ9 no-reward baseline to the synthetic RQ5
first-pilot packet. It does not launch a pilot, choose a real target, select a
live record surface, create a live contribution-log entry, score a
contribution, promise future reward, assign value, adopt reward logic, create
participant rights, change governance, mark tests passed, move claims, change
public posture, or decide any real contribution.

## Purpose

The synthetic first-pilot packet already asks whether a bounded
source-backed research/review contribution could be reviewed without becoming
policy or a real contribution record.

RQ9 adds a narrower pressure:

```text
Can the first-pilot packet preserve recognition, rationale, loss notes, and
contestability without creating reward debt or shadow value?
```

The useful output is not a reward design. The useful output is a no-reward
control screen that a later governed review can inspect before any reward-test
packet is considered.

## Payoff-Variable Reading

Under the T5 bridge, this packet can be read without reward adoption only if
the payoff variables stay narrow.

| variable | first-pilot no-reward reading | hazard if blurred |
|---|---|---|
| `R_c` | Current recognition from a visible draft record and source-checkable rationale. | Recognition becomes status, allocation, review priority, or payout proxy. |
| `L_c` | Legitimacy benefit from a visible process boundary, not from acceptance. | Legitimacy becomes private founder assurance or hidden reviewer discretion. |
| `future_claim(V_c)` | Non-promissory preservation that a later review may inspect the residue. | Residue becomes debt, entitlement, ownership, future eligibility, or score balance. |
| `C_c` | Cost remains hypothetical because no real contributor acted. | Synthetic effort is mistaken for evidence about contributor willingness. |
| `G_d` | Strategic benefit could still come from narrative control or future-value ambiguity. | No-reward status is treated as enough capture protection. |
| `D_d` | Detection would rely on public sources, source/inference separation, and rationale mismatch. | Detection depends on private intuition, hidden AI synthesis, or unstated standards. |
| `S_d` | Existing review states can reject or require revision without sanctions. | Review response becomes punishment, exclusion, or disclosure policy. |
| `F` | Future trust cost is only a durable reasoned record. | The record creates permanent stigma or legal/financial interpretation. |

## Premature Reward-Test Rejection

A later reward-test packet should be rejected as premature if the first-pilot
record cannot answer these questions from public, reviewable material:

1. What visible record would preserve the contribution unit?
2. What recognition is ordinary attribution rather than reward?
3. What rationale is contestable without creating appeal rights?
4. What loss note preserves residue without promising compensation?
5. What language prevents scores, value prompts, or retained records from
   becoming balances?
6. What attack benefit appears even without live payout?
7. Who could pause, revise, or reverse a reward test without changing
   governance?
8. What legal, financial, token, ownership, employment, tax, IP, or securities
   meaning must be excluded from this repo-local scaffold?

If any answer requires private founder context, new policy, economic value
assignment, or external interpretation, the result is:

```text
reward_test_state: rejected_as_premature
```

That result is not a finding that rewards are impossible. It only preserves the
no-reward control boundary.

## Safe Local Repairs

Safe local research can still:

- make no-current-reward language easier to inspect in synthetic packet
  examples;
- separate ordinary attribution from reward, priority, retained-value, and
  governance meanings;
- improve loss-note wording so useful residue does not sound like debt;
- map score-to-reward ambiguity before any score conversion is considered;
- name the attack benefits that remain even without payout;
- or prepare review questions for a later governed RQ9/T5 packet.

Stop for Joe or governed review before:

- adopting a reward mechanism, payout formula, pool, token-like issuance,
  retained-value right, score conversion, review priority, governance weight,
  legal/financial meaning, or public reward posture;
- editing contribution standards, contribution-log schema, issue templates,
  review policy, rights policy, governance, claims, tests, or public-facing
  invitations;
- assigning value to a real or synthetic contribution;
- promising that current records will or will not matter later;
- or taking any non-GitHub external action.
