# RQ2 First-Pilot Class/Evidence Screen

Status: draft review-prep scaffold, not adopted taxonomy, contribution,
review, workflow, governance, rights, reward, public-posture, claim-status, or
contribution-record policy.

This screen applies the existing RQ2/T1 taxonomy scaffolds to the synthetic RQ5
first-pilot packet. It does not launch a pilot, choose a real target, choose a
live record surface, edit templates, change contribution standards, adopt
contribution classes, adopt review paths, assign reviewer authority, mark tests
passed, move claim status, change rights, rewards, governance, public posture,
or create a real contribution record.

## Purpose

The first-pilot queue says that class and evidence continuity are safe local
prep when they stay narrow. The useful question is:

```text
Can one bounded, source-backed research/review contribution shape be classified
and routed without private founder context or hidden policy movement?
```

This file answers only at the review-prep level. It does not decide the first
live pilot target, create an issue or PR, or update active review procedure.

## Synthetic Packet Under Review

This screen uses the same synthetic object as the RQ5 hypothetical
first-pilot packet:

```yaml
packet_id: SAMPLE-RQ5-PACKET-001
primary_shape: bounded source-backed prior-art or mechanism-comparison note
likely_target_area: RQ5/T3/T10/T11 or T9-adjacent review preparation
candidate_primary_class: research
candidate_secondary_effect: review
live_record_surface: undecided
synthetic_review_signal: needs_revision
```

The synthetic signal is not an active review state for a real contribution. It
only records that the packet shape is coherent but still lacks a selected
target, named source system, stable source set, mechanism comparison, and
limitation statement.

## Class Screen

| class question | screen result | why it matters |
|---|---|---|
| What is the likely primary class? | `research`, if the live packet adds a source-backed prior-art or mechanism-comparison note. | The main value is evidence and comparison, not policy adoption. |
| What is the likely secondary effect? | `review`, if the note critiques an existing scaffold, overclaim, review path, or mechanism. | Review value can be preserved without changing the primary class. |
| What should not be treated as primary? | `protocol design`, unless the packet proposes new rules, roles, rights, workflow, or governance movement. | A comparison should not silently become policy. |
| What would make classification unstable? | A bundled rewrite of contribution standards, review states, issue templates, reward meaning, rights, or public posture. | Mixed submissions need narrowing before live review. |
| What is the first safe disposition? | `needs_revision` as a synthetic packet result until target and sources are named. | Missing evidence should not be collapsed into low value, bad faith, or acceptance. |

## Evidence Minimum Screen

The current research-class evidence minimum requires enough public evidence for
a reviewer to evaluate the work without private context.

| evidence field | live packet prompt | current synthetic status |
|---|---|---|
| Source identity | Which system, practice, author, or case is being compared? | Missing. The packet only names the target shape. |
| Stable references | Which stable citations or links support central facts? | Missing. No source packet is attached. |
| Relevance | Which claim, test, document, or research question is affected? | Partially shaped: likely RQ5/T3/T10/T11 or T9-adjacent. |
| Mechanism comparison | Which validation, allocation, voice, contestability, record, or capture mechanism is compared? | Required but not yet answered. |
| Source versus inference | Which facts are sourced, and which claims are contributor interpretation? | Required for a later live packet. |
| Limitation | What uncertainty, non-transferability, or failure mode should prevent overclaim? | Required for a later live packet. |

Synthetic result: the class is plausible, but the evidence packet is not
review-ready. A later live packet should route to revision until these fields
are filled.

## Boundary And Stop Screen

| boundary | safe reading | stop if |
|---|---|---|
| Taxonomy | The packet can use `research` primary and `review` secondary as diagnostic labels. | The run would add, remove, rename, or adopt contribution classes. |
| Review path | The packet can name likely review needs and missing evidence. | The run would adopt review lanes, cadence, appeal rights, response times, or reviewer authority. |
| Contribution standards | The packet can cite existing standards as source context. | The run would edit standards or template fields. |
| Record surface | The packet can stay as a draft project artifact. | The run would open an issue, PR, live log entry, or public pilot surface. |
| Evidence | The packet can preserve missing source, relevance, and limitation fields. | The run would invent sources, use private evidence, or treat thin evidence as accepted. |
| Protected state | The packet can name non-conclusions. | The run would move claims, tests, rights, rewards, governance, AI policy, transfer posture, or public posture. |

## Reviewer Record Shape For Later Use

If a live pilot is authorized later, the class/evidence part of the packet
should preserve these fields before any value scoring or rationale hardens:

| field | review-prep purpose | protected caution |
|---|---|---|
| `primary_class` | Names the main contribution value. | Do not choose the highest-status label by default. |
| `secondary_effect` | Explains a real ambiguity only when needed. | Do not turn every contribution into a multi-class bundle. |
| `evidence_minimum` | Ties review expectations to class. | Do not fill evidence gaps with taste or founder context. |
| `missing_evidence` | Explains why revision is needed. | Do not treat missing fields as proof of bad faith. |
| `classification_rationale` | Lets a later reviewer reconstruct the decision. | A label without rationale is not review evidence. |
| `stop_condition` | Flags governance, rights, reward, public-posture, or evidence-integrity boundaries. | Do not resolve the stop inside ordinary review. |
| `useful_residue` | Preserves useful comparison value if the packet is narrowed. | Do not create reward, retained-value, rights, or future-acceptance meaning. |

## Disagreement Markers

A later reviewer disagreement over this packet should be classified before it
is repaired.

| disagreement | likely meaning | safe next step |
|---|---|---|
| `research` versus `review` ordering | The packet both adds sources and critiques an existing scaffold. | Record primary and secondary class with rationale. |
| `research` versus `synthesis` | The packet integrates multiple repo surfaces but may not add external evidence. | Ask whether stable source evidence or integration value is primary. |
| `research` versus `protocol design` | The packet proposes a rule change while presenting evidence. | Split evidence from policy movement before review continues. |
| `needs_revision` versus `not_accepted` | Reviewers disagree whether missing evidence is repairable. | Name the missing target, source, mechanism comparison, or limitation. |
| `review` versus `adversarial` | A critique may include fabricated or bad-faith evidence. | Separate independently checkable critique from evidence-integrity failure. |

## Useful Residue And Log Eligibility

The synthetic packet has useful residue as a review-prep shape:

- it makes the likely first-pilot contribution class visible;
- it names why `research` is primary only if stable source-backed comparison is
  supplied;
- it preserves `review` as a secondary effect rather than a way to smuggle
  authority;
- it makes missing evidence explicit before value or rationale screens run;
- and it keeps live contribution-log eligibility blocked until a real reviewed
  contribution exists.

This residue does not create reward, retained value, rights, acceptance,
review-policy, governance, claim-status, public-posture, or live-record
meaning.

The packet is not eligible for `CONTRIBUTIONS-LOG.md`.

## What This Prepares

This screen prepares a later human/governed first-pilot review to answer:

1. whether the packet has one reviewable contribution unit;
2. whether `research` is the primary class or only a label for weak evidence;
3. whether `review` is a real secondary effect or hidden policy movement;
4. whether evidence minimums are public enough for review;
5. whether missing evidence routes to revision rather than rejection or
   acceptance;
6. whether useful residue can be preserved without reward or rights language;
7. whether log eligibility remains separate from submission and review state.
